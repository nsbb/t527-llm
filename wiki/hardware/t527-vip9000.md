# T527 Vivante VIP9000-NanoSI-Plus NPU

**Status**: verified
**Last updated**: 2026-07-16
**Related**: [[../pipeline/nbg-export]], [[../pipeline/device-execution]]

## Identity

- SoC: Allwinner T527 (Cortex-A55 × 4 + T527 NPU)
- NPU: Vivante VIP9000-NanoSI-Plus
- Chip ID reported by `vpm_run`: `cid=0x10000016`
- Optimize flag for Acuity export: `VIP9000NANOSI_PLUS_PID0X10000016`
- Class: ~2 TOPS INT8
- Related variant: A733 uses `VIP9000NANODI_PLUS_PID0X1000003B` (petayyyy's target). NBs from A733 target won't load on T527 (`create network 0 failed`).

## Driver stack (on-device)

- `/vendor/lib64/libVIPlite.so` — v1.13 driver
- `/vendor/lib64/libVIPuser.so` — v1.13 user layer
- `/data/local/tmp/vpm_run_aarch64` — test runner binary (deployed 2026-04-01, size 46 KB)
- LD_LIBRARY_PATH must point to `/vendor/lib64` for vpm_run to link

Newer v2.0 stack (`libNBGlinker.so`, `libVIPhal.so`) NOT present on this device.

## Measured performance

| Model | NB | Forward | tok/s pure NPU |
|---|---|---|---|
| SmolLM2-135M | uint8 (124 MB) | 92 ms | 10.9 |
| SmolLM2-135M | int16 dfp (268 MB) | 147 ms | 6.8 |
| SmolLM2-135M | FP32 SW-emul (626 MB) | 7.28 s | 0.14 |
| Qwen2.5-0.5B | uint8 (393 MB) | 198 ms | 5.0 |
| Qwen2.5-0.5B | int16 dfp (891 MB) | 290 ms | 3.4 |
| Qwen2.5-0.5B | FP32 | — | **compile fails** (fatal 64768) |

Overhead (uniform):
- Network create: 200–210 ms (uint8), 650 ms (int16 large)
- Network prepare: 20–30 ms
- Loop overhead: ~1 ms per iteration

adb round-trip adds ~500 ms per token (push input + shell run + pull output). Pure NPU is 5–10× faster than end-to-end.

## Constraints

- **DDR RAM**: 2 GB target (need to confirm actual board)
- **NPU memory pool**: ~1.5 MB for SmolLM2 uint8, ~5 MB for Qwen uint8
- **NBG file storage**: up to ~1 GB works, higher not tested

## Precision support (verified)

| dtype | HW native? | Verdict |
|---|---|---|
| **uint8 / int8 asymmetric_affine** | ✅ native | Fast (92 ms), the target |
| **int8 perchannel_symmetric_affine** | ✅ native | Not tested end-to-end |
| **int16 dynamic_fixed_point** | ⚠ semi-native | Slower (147–290 ms), full HW MAC likely INT8+accumulator INT32 |
| **bfloat16 / qbfloat16** | ❌ NO HW | Acuity NBG export fails with `Fatal 64768` — compiler refuses to emit code for absent HW |
| **float16** | ❌ NO HW | Same as above — VIP9000-NanoSI-Plus has no FP unit |
| **float32** | 🌀 CPU fallback | SmolLM2 FP32 NB (626 MB) works via SW emulation at 7.28 s/forward = **80× slower than uint8**. Qwen FP32 fails (probably NBG size ceiling). Not practical for production. |

**Implication**: **any coherent-generation solution MUST work within uint8/int16 fixed-point on this NPU**. FP path is impractical; wider precision types are the wrong branch to pursue.

## Observed peculiarities

- 3-D input tensor gets promoted to 4-D internally without renumbering attribute axes → see [[../issues/acuity-reducemean-axis]]
- FP32 SW emulation available but ~80× slower than uint8
- Host CPU emulator of quantized ops diverges from device NPU in outlier-heavy layers → see [[../issues/host-vs-device-drift]]
